# 📱 App-Readiness — getting to "ready to develop the iOS/Android app"

> Goal: do **everything except writing the native app**, so that wrapping the
> existing web app with Capacitor is a near-mechanical step. One codebase →
> website + installable PWA + store apps, kept DRY.
> Companion docs: **`PLAN.md`** (overall roadmap), **`HANDOFF.md`**, **`OUTSTANDING.md`**.

## TL;DR — the agreed model
- **The app is the web build.** Capacitor wraps the *same* Svelte SPA inside a native WebView.
  ~90% of code is shared and runs identically; the work is ~10% of **device-level seams**.
- **Guests: no login.** Anonymous `device_id` (already built) + a name + a one-tap
  "notify me". Full order tracking and "your drink" / "how was it?" pushes — **accounts not required**
  (push is keyed to the device, not a user).
- **Staff: real login.** Passwordless staff sign-in (allowlist) replaces the shared PIN `1337`.
- **Notifications are dual-transport:** Web Push (browser/PWA) **and** native APNs/FCM (the app),
  both keyed to the same `device_id`. iOS WebViews can't do Web Push, so the app path *must* be native.
- **The app hard-depends on a public HTTPS endpoint** (the cutover). A phone off the LAN can't reach
  `192.168.1.1`. So **cutover moves onto the critical path.**

## 1. Audit — current state vs. app-ready

### ✅ Already app-friendly (no work needed)
- **Plain Svelte 5 + Vite SPA, no SSR/SvelteKit** — exactly what Capacitor wants.
- **No router, no full-page navigation** — every view is state-driven (`selected`, `showBartender`,
  `orderOpen` in [App.svelte](../apps/web/src/App.svelte)). WebViews hate full reloads; we have none.
- **`viewport-fit=cover` already set** ([index.html:5](../apps/web/index.html:5)) — safe-area groundwork present.
- **Self-hosted fonts + JS confetti** — no CDNs, all bundled (works offline in the wrapper).
- **device-id works in non-secure contexts** ([device.ts](../apps/web/src/lib/device.ts)).
- **Orders already linked to `device_id`** ([db.ts:29](../apps/api/src/db.ts:29)); subscriptions table
  already keyed `(device_id, endpoint)` ([db.ts:33](../apps/api/src/db.ts:33)).
- **npm-workspaces monorepo**, clean scripts, timing-safe key compare, defensive API error handling.

### 🔧 The four seams that must change (the ~10%)
Each becomes a tiny interface with a **web impl** and a **native impl**, chosen at build time.

| Seam | Today | Target |
|---|---|---|
| **API base** | `const BASE = '/api'` hardcoded ([api.ts:12](../apps/web/src/lib/api.ts:12)) | `import.meta.env.VITE_API_BASE ?? '/api'` — web stays same-origin, app points at public HTTPS |
| **Storage** | `localStorage` scattered (device.ts, App.svelte favs, Bartender.svelte key) | `storage.ts` abstraction; native hardens to `@capacitor/preferences` for the durable keys |
| **Auth transport** | Bartender PIN in `X-Bartender-Key` header | One auth server, two transports: **cookie** (web) / **bearer token** (native) |
| **Push transport** | Server plumbing only; no client | Web Push impl (browser) + native APNs/FCM impl (app), dispatched by `transport` |

### ⚙️ Supporting changes
- **CORS** ([config.ts:15](../apps/api/src/config.ts:15)) — `allowedOrigin` is a single value defaulting to `*`.
  Make it a **list** and include `capacitor://localhost` + `https://localhost` (the app origins).
- **Service worker** — `vite-plugin-pwa` registers a Workbox SW. The **native build must NOT register it**
  (the app bundle is its own offline shell; an in-WebView SW caches stale assets). Gate on a build flag.
- **Safe-area CSS** — confirm `env(safe-area-inset-*)` padding on the app shell / tabbar / order rail
  so the notch + home indicator don't overlap UI. `viewport-fit=cover` is already set.

## 2. Data model changes (`packages/shared` + `apps/api/src/db.ts`)
The new app isn't live yet, so we migrate the schema freely. **Phase A shipped the backend-agnostic
columns** (`orders.user_id`, `subscriptions.transport`/`platform`); the **`users`/`sessions` tables land
in Phase B**, generated by the chosen auth library rather than hand-rolled.

```
users        id, display_name, created_at                         -- staff (and future guest accounts)
identities   user_id, provider(passkey|google|apple|email), subject, created_at
sessions     token, user_id, role(guest|bartender), expires_at, created_at
orders       + user_id (nullable)   -- anonymous orders still allowed; links when signed in
subscriptions+ transport(webpush|fcm|apns), platform(web|ios|android)
             -- one table holds web-push subs AND native device tokens, both keyed to device_id
```
- `orders.device_id` stays as the anonymous handle; `user_id` is an optional upgrade.
- `subscriptions.subscription` already stores opaque JSON — fits a native token equally well.

## 3. Auth design
**Recommended foundation: [Better Auth](https://better-auth.com) on the Hono API** — TS-native, self-hosted
on the existing SQLite, no SaaS (matches the "NAS hosts everything" principle). Gives social
(Google/Apple), passkeys, email-OTP, sessions, and anonymous→account linking out of the box.
*(Alt: Firebase Auth — easier native social flows but Google SaaS, against the self-host principle. ☐ DECISION.)*

- **Guests** stay anonymous (no auth touched). If durable guest accounts are ever wanted, add
  passwordless sign-in later and *link* it to the existing `device_id` — no rework.
- **Staff** sign in passwordless (passkey / Apple / Google / email-OTP), gated by a **staff-email
  allowlist** → `bartender` role. Replaces the `requireKey` middleware ([server.ts:78](../apps/api/src/server.ts:78))
  and the PIN gate in [Bartender.svelte](../apps/web/src/lib/Bartender.svelte).
- **Transport:** web uses a session **cookie**; the native WebView uses a **bearer token** stored in
  Preferences (third-party cookies are unreliable in WebViews). Better Auth supports both.
- ⚠️ **App Store rule 4.8:** offering Google on iOS *requires* also offering Sign in with Apple.
- ⚠️ Social OAuth + passkeys need the **public HTTPS domain** (redirect URIs / rpID) → depends on cutover.

## 4. Notification design
Same logic ("tell device X"), two delivery channels picked by `transport`. Keyed to `device_id`, so
**no login required**. Proposed moments (☐ confirm):

| Trigger | Audience | Example |
|---|---|---|
| Order received | guest | (optional — the celebrate UI may suffice) |
| → `making` | guest | 👩‍🍳 "Your Margarita is being made" |
| → `serving` | guest | 🍹 "INCOMING — come grab it!" |
| → `done` (+delay) | guest | "How was it?" → quick rating/feedback |
| New order arrives | bartender | 🔔 "New order from Daniel" |

- Server-side sender lives where the Phase-3 hook is stubbed ([server.ts:113](../apps/api/src/server.ts:113)).
- Web Push needs VAPID keys; native needs an **APNs key (Apple)** + **FCM project (Google)**.
- The OS permission prompt (one tap "Allow") is required and expected — it's consent, not a login.

## 7. Plan & sequence
Checkbox legend: ☐ todo · ◐ in progress · ☑ done

### Phase A — Seams & model ✅ DONE (zero-risk, no external deps)
- ☑ API base via `VITE_API_BASE` ([api.ts](../apps/web/src/lib/api.ts)) — defaults to same-origin `/api`

  the auth library owns its own schema, so hand-rolling them now would be throwaway.
- ☑ CORS `allowedOrigin` → comma-separated list; **prod defaults to the capacitor origins, not `*`**
- ☑ Service worker gated on `VITE_TARGET=native` — verified: the native build emits no `sw.js`/manifest
- ☑ Safe-area — **already** handled in neo.css (appbar/tabbar/order-rail/modals all use `env(safe-area-inset-*)`)

  **Granted subscribe→deliver needs HTTPS** (the cutover) — dormant in prod until then.

- ☐ Staff passwordless login + allowlist → retire PIN gate
- ☐ Confirm notification moments (current: making, serving → guest; new order → bartender)

### Phase C — Human prerequisites (parallel)
- ☐ Cutover: public HTTPS (Cloudflare Tunnel → Caddy), migrate legacy orders, retire PHP
- ☐ Apple Dev + Google Play accounts; Google OAuth client; FCM/APNs keys
